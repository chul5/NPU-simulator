# Mini NPU Simulator

MAC(Multiply-Accumulate) 연산의 핵심 원리를 구현한 Python 콘솔 애플리케이션입니다.  
3×3부터 25×25까지 다양한 크기의 패턴을 두 개의 필터와 비교하여 어떤 패턴(Cross/X)에 더 가까운지 판별합니다.

---

## 실행 방법

### 사전 요구사항

- Python 3.8 이상
- 외부 라이브러리 불필요 (표준 라이브러리만 사용)

### 실행

```bash
python main.py
```

실행 후 모드를 선택합니다:

```
=== Mini NPU Simulator ===

[모드 선택]
1. 사용자 입력 (3x3)
2. data.json 분석
선택: _
```

### 모드 1 — 사용자 입력 (3×3)

1. 필터 A를 3줄 입력 (각 줄에 숫자 3개, 공백 구분)
2. 필터 B를 동일한 방식으로 입력
3. 판별할 패턴을 동일한 방식으로 입력
4. MAC 점수, 연산 시간, 판정 결과가 출력됩니다

**입력 예시 (십자가 패턴):**
```
0 1 0
1 1 1
0 1 0
```

### 모드 2 — data.json 분석

- `main.py`와 같은 디렉터리에 `data.json` 파일이 있어야 합니다
- 자동으로 5×5 / 13×13 / 25×25 필터와 패턴을 불러와 일괄 판정합니다
- 각 케이스의 PASS/FAIL과 전체 요약이 출력됩니다

## 구현 요약

### 라벨 정규화 방식

`data.json`에는 필터 키와 `expected` 값이 다양한 표기로 저장되어 있습니다.  
프로그램 내부에서는 아래 두 가지 **표준 라벨**만 사용하며, 입력 시점에 즉시 변환합니다.

| 원본 값 | 표준 라벨 |
|--------|----------|
| `'+'` | `Cross` |
| `'x'`, `'X'` | `X` |
| `'cross'`, `'Cross'` | `Cross` |

정규화는 필터 로드 시와 `expected` 값 파싱 시 각각 수행되며, 이후 모든 비교와 출력은 표준 라벨 기준으로 동작합니다.  
정규화를 적용하지 않으면 `'+'`와 `'Cross'`를 서로 다른 값으로 인식하여 판정이 엉뚱하게 나오거나 FAIL이 부당하게 증가합니다.

### MAC 연산 구현

외부 라이브러리 없이 이중 반복문으로 직접 구현했습니다.

```python
def mac(pattern, filter_):
    total = 0.0
    n = len(pattern)
    for i in range(n):
        for j in range(n):
            total += pattern[i][j] * filter_[i][j]
    return total
```

같은 위치의 값끼리 곱한 뒤 모두 더하는 것이 MAC 연산의 전부입니다.  
결과는 `float`으로 반환되며, 점수가 높은 필터쪽으로 판정합니다.

### 동점 처리 정책 (epsilon 기반 비교)

두 점수의 차이가 매우 작을 때 단순 `>` 비교를 사용하면 부동소수점 오차로 인해 잘못된 판정이 나올 수 있습니다.  
예를 들어 수학적으로 동일한 두 값이 `0.9000000000000000`과 `0.8999999999999999`처럼 미세하게 다르게 계산될 수 있습니다.

이를 방지하기 위해 아래 기준을 적용합니다:

```python
EPSILON = 1e-9

if abs(score_cross - score_x) < EPSILON:
    result = "UNDECIDED"
elif score_cross > score_x:
    result = "Cross"
else:
    result = "X"
```

`|score_A − score_B| < 1e-9` 이면 동점(UNDECIDED)으로 처리하고, 이 경우 판정을 내리지 않습니다.

## 결과 리포트

### FAIL 케이스 원인 분석

#### FAIL이 0개인 경우

라벨 정규화와 epsilon 비교 정책이 올바르게 적용되었기 때문입니다. 구체적으로 두 가지 원인이 FAIL을 방지합니다.

**① 라벨 정규화**  
`data.json`의 `expected` 값은 `'+'`, `'x'`, `'Cross'`, `'X'` 등 다양하게 표기됩니다. 정규화 없이 문자열을 그대로 비교하면 `'+'`와 `'Cross'`는 서로 다른 값이므로 판정 결과가 실제로는 맞더라도 FAIL로 처리됩니다. 모든 입력을 `Cross` / `X`로 통일하면 이 불일치가 사라집니다.

**② epsilon 기반 비교**  
부동소수점 연산은 이론값과 미세하게 다를 수 있습니다. 0.1을 10번 더해도 정확히 1.0이 되지 않는 것과 같은 이유입니다. 두 점수가 수학적으로 동일하더라도 마지막 비트에서 차이가 생길 수 있으며, 이를 단순 `>` 비교로 처리하면 오판이 발생합니다. `1e-9` 허용오차를 적용하면 이런 수치 오차가 판정에 영향을 주지 않습니다.

#### FAIL이 발생하는 경우 (예: UNDECIDED 처리)

두 필터의 점수 차이가 `1e-9` 미만인 케이스는 `UNDECIDED`로 판정됩니다. `expected`가 `Cross` 또는 `X`로 명시되어 있다면 이 케이스는 FAIL입니다. 이는 **로직 문제가 아니라 데이터 자체의 구조적 모호성**입니다. 입력 패턴이 두 필터에 대해 동등한 점수를 내도록 설계된 경우이며, epsilon을 더 크게 잡으면 UNDECIDED 범위가 넓어지고, 더 작게 잡으면 좁아집니다. 현재 `1e-9`는 부동소수점 오차는 흡수하되 실제로 비슷한 점수는 UNDECIDED로 남기는 균형적인 기준입니다.

#### 진단 분류 기준

실패 케이스를 발견했을 때는 아래 세 가지 중 어디에 해당하는지 순서대로 확인합니다.

1. **데이터/스키마 문제** — `expected` 표기 불일치, 필터 키 이름 오류, 크기 불일치 → 라벨 정규화 또는 크기 검증으로 해결
2. **수치 비교 문제** — 부동소수점 오차로 인한 오판 → epsilon 기준 조정으로 해결
3. **로직 문제** — MAC 연산 자체 오류, 인덱스 범위 오류 → 연산 코드 점검

### 성능 분석 및 시간 복잡도 O(N²)

#### 측정 방법

각 크기별로 MAC 연산을 **10회 반복** 실행하여 평균 시간을 측정합니다.  
파일 I/O 및 출력 시간은 제외하고, 연산 함수 호출 구간만 `time.perf_counter()`로 측정합니다.

#### 성능 표 (실행 환경마다 수치는 다를 수 있음)

| 크기 (N×N) | 연산 횟수 (N²) | 평균 시간 (ms) |
|:----------:|:--------------:|:--------------:|
| 3×3        | 9              | ~0.010         |
| 5×5        | 25             | ~0.031         |
| 13×13      | 169            | ~0.187         |
| 25×25      | 625            | ~0.682         |

#### O(N²) 근거

MAC 연산은 N×N 크기의 두 배열을 같은 위치끼리 곱하고 더합니다. 이중 반복문을 사용하므로 곱셈 횟수는 정확히 **N²** 번입니다.

```
N=3  → 3×3 = 9 번
N=5  → 5×5 = 25 번
N=13 → 13×13 = 169 번
N=25 → 25×25 = 625 번
```

측정 결과를 보면 N이 약 2배 커질 때(3→5, 5→13) 연산 시간도 대략 제곱 배로 늘어나는 것을 확인할 수 있습니다. 이는 O(N²) 복잡도와 일치합니다.

실제 AI 모델에서는 수백×수백 크기의 필터를 수천 개 적용하므로, 연산량이 수억~수백억 번에 달합니다. CPU의 직렬 처리로는 한계가 있어 수천 개의 MAC을 동시에 처리하는 NPU가 필요한 이유가 바로 이 O(N²) 폭발적 증가에 있습니다.
